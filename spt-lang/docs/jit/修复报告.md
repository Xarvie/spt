## 1. 卡死的根因（改动 #4 到底错在哪）

改动 #4 想做的事：把循环不变的 `t->array` 指针在 preheader 里加载到一个专用寄存器（如 R9），
循环体里所有读同一数组的 GETI 直接复用这个寄存器，省掉每次的 spill 读取 + `t->array` 解引用。

但它和**已有的 LICM（循环不变式外提）**发生了致命冲突：

1. `arr_read_int.spt` 的内层循环 `for(int j=0,7)` 会被**展开**，于是 8 个 `a[j]` 全变成
   **常量下标** `a[0]..a[7]`。
2. `ra_hoist_invariants()` 会把**常量下标的 GETI 整条外提到 preheader**（代码里明确允许，
   见 LICM 的 `SPTIR_GETI` 分支）。
3. 但 preheader 的发射顺序是：**先发射所有被外提的指令（含这些 GETI），最后才加载缓存寄存器**
   （`sptjit_codegen_compile` 里的 preheader 块）。
4. 而被外提的 GETI 在 codegen 时会执行 `mov RAX, arr_cache_reg`——此时 `arr_cache_reg`(R9)
   **还没被赋值**。`gen_prologue` 只初始化了 R12/R13/RBX/R14，从不碰 R8–R11，所以 R9 里是
   调用方残留的**垃圾值**。
5. GETI 用这个垃圾指针去读元素标签 → 标签守卫失败 → **每次进入 trace 都立即 side-exit** →
   反复重编译 → 运行时拉黑（`side_exits == entry_count == 10240`，`runtime_fails` 1→2→3）→
   表现为**卡死/活锁**。

> 你保存的顶层 `jit_debug.txt`（`pc_offset=17 side_exits=10240 entry_count=10240`）和我复现到的
> 现象完全一致，确认根因定位正确。

**关键观察**：当所有 GETI 都被外提后，循环体里**根本没有 GETI 再读这个缓存寄存器**——
也就是说，对这个 benchmark 而言，改动 #4 既**没用**又**有害**。这正是交接文档里你猜测的方向。

---

## 2. 为什么选择「移除 #4」而不是「修好 #4」

我先实现了一个能跑通的修复版（让被外提的 GETI 不走缓存分支），全套测试都绿。
但随后做了 A/B 性能基准，结论让 #4 失去了保留价值：

**变量下标数组读取**（`s = s + a[j]`，j 为循环携带变量，4000 万次；这正是 #4 设计的目标场景）：

| 配置 | 最优耗时 |
|------|---------|
| 解释器 (INT) | 766–794 ms |
| JIT，#4 关闭 | **261 ms** |
| JIT，#4 开启 | 263 ms |

- JIT 现在比解释器**快约 2.9×**——交接文档「数组读取比解释器慢 2×」的问题，靠 **#1–#3 已经解决**。
- **#4 带来的差异是 0（261 vs 263ms，在噪声范围内）**。原因：#2 的 SIB 寻址已经让元素访问足够便宜，
  而 `t->array` 这一次加载本就命中 L1，省下它毫无收益。
- 标量对照（`s = s + i`，4000 万次）：INT 290 ms → JIT 28 ms，约 10×，说明 JIT 主体是健康的。

既然 #4 **引入了严重卡死、又证明零收益**，还白占一个寄存器、增加 codegen 复杂度，
**最稳妥的工程选择就是整条移除**（也正是交接文档给出的备选方案）。
改动后 JIT 数组读取性能与「修好 #4」完全一致。

## 3. 具体改了什么

只动了一个文件：`src/jit/spt_jit_codegen.c`，纯删除，共 5 处（详见 `remove_change4.patch`）：

1. `SPTCodeGen` 结构体里的 `arr_cache_ref` / `arr_cache_reg` 两个字段
2. 整个 `ra_setup_arr_cache()` 函数
3. `sptjit_codegen_compile` 里对 `ra_setup_arr_cache(&cg)` 的调用
4. preheader 里的缓存加载块
5. GETI codegen 里的缓存分支（恢复为 #4 之前的两行 `gen_load + 解引用`）

**#1、#2、#3 三项改动原样保留、未作改动**——它们已验证正确，是性能提升的真正来源。

净变化：1951 行 → 1864 行（-87 行）。

## 4. 顺带发现的一个旧 bug（不在本次范围，供你参考）

`nested_histogram.spt`（`a[i+j+k]++` 同数组读写碰撞的回归守卫）在 **JIT 开启**时输出错误：

```
正确(off): 200001|600003|1200006|2000010|2400012|2400012|2000010|1200006|600003|200001
错误(on) : 200002|600006|1200012|2000020|12|12|10|6|3|200002
```

**这是改动 #4 之前就存在的旧 bug，与本次工作无关**，我已用未改动的原始二进制验证：原始版输出
完全相同的错误结果。原因也很清楚：该 kernel 写数组，`ra_setup_arr_cache` 本来就会因 `SETI` 提前
返回，#4 对它根本不生效。

- 它**不影响 ctest**：ctest 默认 JIT 关闭，此用例在解释器下输出正确，仍计入 372/372。
- 它只在 JIT 差分测试里暴露。你之前的差分脚本没有超时，会先卡在 `arr_read_int` 上，所以一直没跑到它。
- 该 kernel 注释说「展开器应当拒绝展开同时读写同一数组的循环体」，说明**这条拒绝逻辑存在漏洞/回归**。
  这是一个独立的 JIT 正确性问题，建议单独开 issue 处理，**不要**和本次提交混在一起。

## 6. 在本机（Linux，无 cmake/ninja）如何构建与测试

本机只有 gcc + make，没有 cmake/ninja，也没有网络，所以我用 gcc 直接手动构建（与 CMakeLists 一致：
`spt_core = vm + frontend + codegen + jit`，再加 cli）。附带的脚本：

- `build.sh` —— 一行 gcc 构建出 `build_manual/sptscript`
- `run_spt_tests.sh <绝对路径bin> off` —— 跑 ctest 等价的 .spt 套件（注意 bin 要用**绝对路径**）
- `jit_difftest_to.sh <绝对路径bin>` —— JIT 开/关差分测试（带超时，含十亿次 benchmark 的处理）
- `timeit.sh` —— 简单计时器

> 注：仓库里的 `test_arr_flat.spt` 开头有 BOM（U+FEFF），会导致解析失败，需要时删掉 BOM 即可。
> `sptxx/` 的 C++ 测试不在压缩包内，已忽略（与本任务无关）。
